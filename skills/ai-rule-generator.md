# AI 不足捕捉与规则生成器

## 功能说明

> **核心目标**：在 AI 与用户的协作过程中，自动捕捉 AI 表现出的不足之处（如错误的代码风格、遗漏的边界条件、不符合项目规范的行为等），分析根因，并自动生成对应的规则条目写入规则文件，形成**自我进化的规则体系**，持续提升 AI 的协作质量。

本 Skill 的工作模式是**观察者 + 规则工厂**——在对话过程中持续监控 AI 的表现，当检测到不足时自动触发规则生成流程。

### 核心能力

1. **不足捕捉**：自动识别 AI 在对话中暴露的问题和不足
2. **根因分析**：深入分析不足的根本原因，而非表面现象
3. **规则生成**：将分析结果转化为结构化、可执行的规则条目
4. **规则归档**：将规则写入正确的规则文件，并更新索引
5. **去重校验**：避免生成与已有规则重复或冲突的条目

## AI 执行流程

### Step 1：捕捉不足信号

AI 应在对话过程中持续监控以下**不足信号**：

#### 1.1 显式信号（用户主动反馈）

| 信号类型 | 示例 | 严重程度 |
|---------|------|---------|
| 直接纠正 | "不对，应该用 XXX 而不是 YYY" | 🔴 高 |
| 否定反馈 | "这不是我要的"、"不行"、"错了" | 🔴 高 |
| 补充要求 | "你漏了 XXX"、"还需要考虑 YYY" | 🟡 中 |
| 风格偏好 | "以后不要用这种写法"、"我习惯用 XXX" | 🟡 中 |
| 重复提醒 | 用户多次提到同一个要求 | 🔴 高 |

#### 1.2 隐式信号（AI 自我检测）

| 信号类型 | 检测方式 | 严重程度 |
|---------|---------|---------|
| 代码被大量修改 | 用户对 AI 生成的代码做了大幅改动 | 🟡 中 |
| 反复追问 | 用户对同一问题追问 3 次以上 | 🔴 高 |
| 回退操作 | 用户要求撤销 AI 的修改 | 🔴 高 |
| 与项目风格不一致 | AI 生成的代码与项目现有风格明显不同 | 🟡 中 |
| 遗漏边界条件 | AI 未处理空值、异常、并发等边界情况 | 🟡 中 |
| 违反已有规则 | AI 的行为违反了 rules/ 目录下已有的规则 | 🔴 高 |

**触发阈值**：
- 🔴 高严重度信号：**立即触发**规则生成流程
- 🟡 中严重度信号：**累计 2 次**同类信号后触发
- 同一对话中同一类问题只触发一次规则生成

### Step 2：分析不足根因

捕捉到不足信号后，进行根因分析：

```
根因分析框架：
├── 1. 问题描述：AI 具体做错了什么？
├── 2. 期望行为：正确的做法应该是什么？
├── 3. 根因分类：
│   ├── 📌 知识缺失：AI 不知道某个规范/约定/最佳实践
│   ├── 📌 上下文遗漏：AI 未充分理解项目上下文
│   ├── 📌 规则缺失：现有规则体系中没有覆盖此场景
│   ├── 📌 规则模糊：现有规则描述不够明确，导致执行偏差
│   └── 📌 优先级错误：AI 在多个规则冲突时选择了错误的优先级
├── 4. 影响范围：此问题是个例还是会反复出现？
└── 5. 规则建议：应该生成什么样的规则来防止此问题？
```

### Step 3：校验规则必要性

在生成规则之前，必须进行以下校验：

#### 3.1 去重检查

读取 `rules/index.md` 和相关规则文件，确认：

- [ ] 不存在已有规则覆盖此场景
- [ ] 不存在与新规则冲突的已有规则
- [ ] 如果已有规则部分覆盖，应选择**更新已有规则**而非新建

#### 3.2 通用性评估

评估规则的通用性，决定规则的归属：

| 通用性 | 归属 | 示例 |
|--------|------|------|
| 跨项目通用 | `rules/common/` 下新建或追加 | "所有 API 必须有错误处理" |
| 语言特定 | `rules/languages/{lang}/` 下新建或追加 | "Go 函数参数不超过 4 个" |
| 项目特定 | 项目内 `.ai/` 或 `.codebuddy/rules/` 下 | "本项目使用 gorm 而非 sqlx" |
| 个人偏好 | 使用 memory 存储，不写入规则文件 | "用户喜欢简洁的注释风格" |

#### 3.3 质量门槛

新规则必须满足以下质量标准：

- [ ] **可执行**：规则描述足够具体，AI 可以直接执行，无歧义
- [ ] **可验证**：能够通过代码审查或自动检查验证是否遵守
- [ ] **有示例**：包含 ✅ 正确和 ❌ 错误的代码示例（如适用）
- [ ] **有边界**：明确规则的适用范围和例外情况

### Step 4：生成规则条目

根据分析结果，生成规则条目。规则条目必须遵循以下格式：

#### 4.1 追加到已有规则文件

如果规则属于已有规则文件的范畴，追加到对应文件末尾：

```markdown
### {规则名称}（{等级}）

- 规则要点 1
- 规则要点 2
- 规则要点 3

```{language}
// ✅ 正确
正确示例代码

// ❌ 错误
错误示例代码
```
```

> 此格式遵循 `rules/common/rule.md` 中「追加条目书写格式（强制）」的要求。

#### 4.2 创建新规则文件

如果规则不属于任何已有文件的范畴，创建新的规则文件：

```markdown
# {规则名称}

> **{等级}**：{一句话描述规则的核心要求}

## 1. {章节标题}

### 1.1 {子章节}
- 规则要点
- 规则要点

```{language}
// ✅ 正确
正确示例代码

// ❌ 错误
错误示例代码
```

## 2. {章节标题}
...
```

### Step 5：写入规则文件并更新索引

#### 5.1 写入规则

- **追加模式**：将规则条目追加到目标规则文件的指定位置
- **新建模式**：在 `rules/` 对应子目录下创建新的 `.md` 文件

#### 5.2 更新 rules/index.md

按照 `rules/index.md` 中的索引维护指令，同步更新索引：

- 新增规则文件：在「快速导航」和「详细索引」中添加对应条目
- 修改已有规则文件：更新受影响的条目描述和日期

#### 5.3 更新 skills/index.md（如涉及）

如果规则变更影响了 Skill 的行为描述，同步更新 Skill 索引。

### Step 6：输出不足分析报告

向用户展示规则生成结果：

```
🔍 AI 不足捕捉报告
━━━━━━━━━━━━━━━━━━
📌 发现问题：{问题描述}
🔎 根因分析：{根因分类} — {详细分析}
📏 生成规则：{规则名称}（{等级}）
📁 写入位置：{文件路径}
📝 规则内容摘要：
   - {要点 1}
   - {要点 2}
   - {要点 3}
━━━━━━━━━━━━━━━━━━
💡 此规则将在后续对话中自动生效
```

## 不足分类与规则映射

### 常见不足类型及对应规则方向

| 不足类型 | 典型表现 | 规则方向 | 目标文件 |
|---------|---------|---------|---------|
| 代码风格不一致 | 命名风格、缩进、注释格式与项目不符 | 编码规范补充 | `rules/languages/{lang}/code-style.md` |
| 错误处理不当 | 吞掉错误、错误分类不准确 | 错误处理规范补充 | `rules/common/error-handling.md` |
| API 设计不规范 | 路由命名不统一、响应格式不一致 | API 设计规范补充 | `rules/common/api-design.md` |
| 数据库操作不安全 | SQL 拼接、缺少事务、N+1 查询 | 数据库规范补充 | `rules/common/database.md` |
| 架构违规 | 越层调用、逆向依赖 | 架构约束补充 | `rules/common/rule.md` |
| 上下文理解不足 | 未读取关键文件就开始编码 | 上下文管理规则补充 | `rules/common/rule.md` |
| 边界条件遗漏 | 未处理空值、并发、超限等 | 质量自检清单补充 | `rules/common/rule.md` |
| 安全问题 | XSS、注入、敏感信息泄露 | 安全规范（新建） | `rules/common/security.md` |
| 性能问题 | 无分页、无缓存、重复查询 | 性能规范（新建） | `rules/common/performance.md` |
| 测试不足 | 缺少单测、测试覆盖不全 | 测试规范（新建） | `rules/common/testing.md` |

| 指令 | 说明 |
|------|------|

## 规则生成示例

### 示例 1：用户纠正代码风格

**对话场景**：
```
用户：帮我写一个获取用户列表的接口
AI：（生成了使用 `self` 作为方法接收器的代码）
用户：不要用 self，Go 里面用类名首字母小写
```

**捕捉到的信号**：直接纠正（🔴 高）

**生成的规则**（追加到 `rules/languages/go/code-style.md`）：

```markdown
### 方法接收器命名（强制）

- 方法接收器使用类名首字母小写，禁止使用 `self`/`this`/`me`
- 接收器名称应简短且有意义

```go
// ✅ 正确
func (u *User) GetName() string {
    return u.Name
}

// ❌ 错误
func (self *User) GetName() string {
    return self.Name
}
```
```

### 示例 2：AI 遗漏错误处理

**对话场景**：
```
用户：帮我写一个文件读取函数
AI：（生成了没有处理 os.Open 错误的代码）
用户：你怎么没处理错误？
```

**捕捉到的信号**：补充要求（🟡 中）+ 违反已有规则（🔴 高）

**生成的规则**（追加到 `rules/common/error-handling.md`）：

```markdown
### 文件操作错误处理（强制）

- 所有文件操作（Open/Read/Write/Close）必须检查并处理错误
- 文件打开后必须使用 defer 确保关闭
- 错误信息应包含文件路径，便于排查

```go
// ✅ 正确
f, err := os.Open(filepath)
if err != nil {
    return fmt.Errorf("打开文件失败 %s: %w", filepath, err)
}
defer f.Close()

// ❌ 错误
f, _ := os.Open(filepath)
```
```

### 示例 3：发现新的规范领域

**对话场景**：
```
用户：帮我写一个处理用户输入的接口
AI：（生成了没有输入校验和 XSS 防护的代码）
用户：这样不安全吧，输入都没校验
```

**捕捉到的信号**：否定反馈（🔴 高）

**生成的规则**（新建 `rules/common/security.md`）：

```markdown
# 安全规范

> **强制**：所有涉及用户输入的代码必须遵守本文件中的安全规范

## 1. 输入校验

### 1.1 通用校验规则
- 所有用户输入必须进行长度限制
- 字符串输入必须进行特殊字符过滤或转义
- 数值输入必须校验范围
- 禁止直接将用户输入拼接到 SQL/HTML/Shell 命令中
...
```

## 注意事项

1. **最小侵入原则**：规则生成不应打断用户的主要工作流，分析和生成在后台完成，仅在完成后简要通知用户
2. **精准定位原则**：规则应写入最合适的文件位置，优先追加到已有规则文件，避免规则碎片化
3. **避免过度生成**：不是所有纠正都需要生成规则，一次性的、项目特定的偏好应使用 memory 存储
4. **用户确认原则**：对于 🔴 高严重度的新规则，建议在写入前向用户确认；对于追加到已有规则的条目，可直接写入
5. **规则一致性**：新生成的规则必须与已有规则体系保持一致，格式、用语、等级标注都要统一
6. **可追溯性**：规则的变更历史由 git log 追踪，规则文件中不单独记录变更日志
7. **定期回顾**：建议用户定期使用 `#rule-gen report` 查看累积的规则，清理不再适用的条目
