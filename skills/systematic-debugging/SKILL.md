# 系统化调试

## 功能说明

> **核心目标**：随机修复浪费时间并制造新 Bug。快速补丁掩盖底层问题。**必须先找到根因，再尝试修复。症状修复 = 失败。**

**违反此流程的字面意思就是违反调试的精神。**

---

## 铁律

```
没有根因调查 → 不提出修复方案
```

如果你没有完成阶段一，你**不能**提出修复方案。

---

## 适用场景

用于**任何**技术问题：
- 测试失败
- 生产环境 Bug
- 意外行为
- 性能问题
- 构建失败
- 集成问题

**尤其在以下情况使用：**
- 时间压力下（紧急情况让猜测变得诱人）
- "就一个快速修复"看起来很明显
- 你已经尝试了多个修复
- 上一个修复没有生效
- 你不完全理解问题

**不要跳过，当：**
- 问题看起来很简单（简单 Bug 也有根因）
- 你很赶时间（系统化比瞎折腾更快）
- 领导要求立即修复（系统化比反复返工更快）

---

## 四个阶段

你**必须**完成每个阶段后才能进入下一个。

### 阶段一：根因调查

**在尝试任何修复之前：**

#### 1. 仔细阅读错误信息

- 不要跳过错误或警告
- 它们通常包含确切的解决方案
- 完整阅读堆栈跟踪
- 记录行号、文件路径、错误代码

#### 2. 稳定复现

- 能可靠地触发吗？
- 确切的步骤是什么？
- 每次都发生吗？
- 如果不能复现 → 收集更多数据，不要猜测

#### 3. 检查最近变更

- 什么变更可能导致了这个问题？
- Git diff，最近的提交
- 新依赖、配置变更
- 环境差异

#### 4. 在多组件系统中收集证据

**当系统有多个组件时（CI → 构建 → 签名，API → 服务 → 数据库）：**

**在提出修复之前，添加诊断工具：**

```
对每个组件边界：
  - 记录进入组件的数据
  - 记录离开组件的数据
  - 验证环境/配置传播
  - 检查每层的状态

运行一次收集证据，展示哪里断了
然后分析证据，识别失败的组件
然后调查那个特定组件
```

#### 5. 追踪数据流

**当错误在调用栈深处时：**

- 坏值从哪里来？
- 谁用坏值调用了这个？
- 继续向上追踪直到找到源头
- 在源头修复，不是在症状处

### 阶段二：模式分析

**在修复之前找到模式：**

1. **找到工作的示例** — 在同一代码库中找到类似的工作代码
2. **对比参考** — 如果在实现某个模式，完整阅读参考实现
3. **识别差异** — 工作的和坏的之间有什么不同？列出每个差异
4. **理解依赖** — 这需要什么其他组件？什么设置、配置、环境？

### 阶段三：假设与测试

**科学方法：**

1. **形成单一假设** — 清楚地陈述："我认为 X 是根因，因为 Y"
2. **最小测试** — 做最小的可能变更来测试假设。一次一个变量
3. **验证后再继续** — 有效？→ 阶段四。无效？→ 形成新假设。**不要**在上面叠加更多修复
4. **不知道时** — 说"我不理解 X"。不要假装知道

### 阶段四：实现

**修复根因，不是症状：**

#### 1. 创建失败的测试用例

- 最简单的复现
- 如果可能，自动化测试
- 使用 `test-driven-development` Skill 编写正确的失败测试
- 修复前**必须**有测试

#### 2. 实现单一修复

- 解决已识别的根因
- **一次一个**变更
- 没有"顺便"的改进
- 没有捆绑的重构

#### 3. 验证修复

- 测试现在通过了？
- 其他测试没有坏？
- 问题真的解决了？

#### 4. 如果修复无效

- **停下**
- 数一数：你尝试了多少次修复？
- 如果 < 3：回到阶段一，用新信息重新分析
- **如果 ≥ 3：停下并质疑架构（见下方第 5 步）**
- **不要**在没有架构讨论的情况下尝试第 4 次修复

#### 5. 如果 3+ 次修复失败：质疑架构

**表明架构问题的模式：**
- 每次修复揭示新的共享状态/耦合/不同位置的问题
- 修复需要"大规模重构"才能实现
- 每次修复在其他地方产生新症状

**停下并质疑根本：**
- 这个模式从根本上合理吗？
- 我们是否因为惯性而坚持？
- 应该重构架构还是继续修复症状？

**在尝试更多修复之前与用户讨论。**

---

## 危险信号 — 立即停下并回到阶段一

如果你发现自己在想：
- "先快速修复，后面再调查"
- "试试改一下 X 看看行不行"
- "加多个修改，跑测试"
- "跳过测试，我手动验证"
- "大概是 X，让我修一下"
- "我不完全理解但这可能有效"
- "模式说 X 但我换个方式"
- "主要问题是：[列出修复方案但没有调查]"
- 在追踪数据流之前就提出解决方案
- **"再试一次修复"（已经试了 2+ 次）**
- **每次修复揭示不同位置的新问题**

**以上所有都意味着：停下。回到阶段一。**

**如果 3+ 次修复失败：** 质疑架构。

## 常见借口反驳表

| 借口 | 现实 |
|------|------|
| "问题很简单，不需要流程" | 简单问题也有根因。流程对简单 Bug 很快 |
| "紧急情况，没时间走流程" | 系统化调试比猜测-检查-瞎折腾**更快** |
| "先试这个，然后再调查" | 第一次修复设定了模式。从一开始就做对 |
| "确认修复有效后再写测试" | 未测试的修复不持久。先写测试证明它 |
| "一次修多个节省时间" | 无法隔离什么有效。会引入新 Bug |
| "参考太长，我改编一下" | 部分理解保证有 Bug。完整阅读 |
| "我看到问题了，让我修" | 看到症状 ≠ 理解根因 |
| "再试一次修复"（2+ 次失败后） | 3+ 次失败 = 架构问题。质疑模式，不要再修 |

## 快速参考

| 阶段 | 关键活动 | 成功标准 |
|------|---------|---------|
| **1. 根因** | 读错误、复现、检查变更、收集证据 | 理解**什么**和**为什么** |
| **2. 模式** | 找工作示例、对比 | 识别差异 |
| **3. 假设** | 形成理论、最小测试 | 确认或新假设 |
| **4. 实现** | 创建测试、修复、验证 | Bug 解决，测试通过 |

---

## 真实世界影响

来自调试会话的数据：
- 系统化方法：15-30 分钟修复
- 随机修复方法：2-3 小时瞎折腾
- 首次修复成功率：95% vs 40%
- 引入新 Bug：接近零 vs 常见
