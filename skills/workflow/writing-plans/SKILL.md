# 实现计划编写

## 功能说明

> **核心目标**：编写全面的实现计划，假设执行者对代码库零上下文、品味存疑。记录他们需要知道的一切：每个任务要修改哪些文件、完整代码、测试方法、需要查阅的文档、如何验证。将整个计划拆分为一口大小的任务。DRY、YAGNI、TDD、频繁提交。

假设执行者是一个技能熟练的开发者，但对我们的工具集和问题领域几乎一无所知。假设他们不太擅长测试设计。

---

## 铁律

```
没有计划的实现 = 没有地图的旅行
每个步骤必须包含实际内容，禁止占位符
```

---

## 范围检查

如果设计覆盖了多个独立子系统，它应该在 brainstorming 阶段就被拆分为子项目设计。如果没有，建议拆分为独立的计划——每个子系统一个。每个计划应该能独立产出可工作、可测试的软件。

---

## 文件结构规划

在定义任务之前，先规划哪些文件将被创建或修改，以及每个文件负责什么。这是分解决策被锁定的地方。

- 设计边界清晰、接口定义良好的单元。每个文件应该有一个明确的职责
- AI 在能一次性放入上下文的代码上推理更好，文件聚焦时编辑更可靠。优先选择更小、聚焦的文件
- 一起变更的文件应该放在一起。按职责拆分，而非按技术层拆分
- 在已有代码库中，遵循已有模式

---

## 任务粒度：一口大小

**每个步骤是一个动作（2-5 分钟）：**
- "编写失败的测试" — 一个步骤
- "运行测试确认失败" — 一个步骤
- "编写最小代码使测试通过" — 一个步骤
- "运行测试确认通过" — 一个步骤
- "提交" — 一个步骤

## 计划文档头部

**每个计划必须以此头部开始：**

```markdown
# [功能名称] 实现计划

> **执行指引：** 使用 `subagent-driven-development` Skill（推荐）或逐步执行此计划。步骤使用 checkbox（`- [ ]`）语法跟踪进度。

**目标：** [一句话描述要构建什么]

**架构：** [2-3 句话描述方案]

**技术栈：** [关键技术/库]

---
```

## 任务结构模板

````markdown
### 任务 N：[组件名称]

**文件：**
- 创建：`exact/path/to/file.go`
- 修改：`exact/path/to/existing.go:123-145`
- 测试：`tests/exact/path/to/test.go`

- [ ] **步骤 1：编写失败的测试**

```go
func TestSpecificBehavior(t *testing.T) {
    result := Function(input)
    assert.Equal(t, expected, result)
}
```

- [ ] **步骤 2：运行测试确认失败**

运行：`go test ./path/to/... -run TestSpecificBehavior -v`
预期：FAIL，提示 "undefined: Function"

- [ ] **步骤 3：编写最小实现**

```go
func Function(input Type) ReturnType {
    return expected
}
```

- [ ] **步骤 4：运行测试确认通过**

运行：`go test ./path/to/... -run TestSpecificBehavior -v`
预期：PASS

- [ ] **步骤 5：提交**

```bash
git add path/to/test.go path/to/file.go
git commit -m "feat: add specific feature"
```
````

## 禁止占位符

每个步骤必须包含工程师需要的实际内容。以下是**计划失败**——永远不要写：

| 禁止内容 | 为什么是失败 |
|---------|------------|
| "TBD"、"TODO"、"后续实现" | 执行者不知道该做什么 |
| "添加适当的错误处理" | "适当"是什么？给出具体代码 |
| "编写上述代码的测试" | 没有实际测试代码 |
| "类似任务 N" | 执行者可能不按顺序读任务，重复代码 |
| 描述做什么但不展示怎么做的步骤 | 代码步骤必须有代码块 |
| 引用未在任何任务中定义的类型/函数 | 执行者找不到定义 |

## 计划自审

写完完整计划后，用全新的眼光对照设计文档检查：

1. **设计覆盖度**：浏览设计文档的每个章节/需求。能否指出实现它的任务？列出差距
2. **占位符扫描**：搜索上方"禁止占位符"中的所有模式。修复它们
3. **类型一致性**：后面任务中使用的类型、方法签名、属性名是否与前面任务中定义的一致？任务 3 中叫 `ClearLayers()` 但任务 7 中叫 `ClearFullLayers()` 就是 bug

发现问题就直接修复。如果发现设计需求没有对应任务，添加任务。

## 执行交接

保存计划后，提供执行选择：

```
计划已完成并保存到 `docs/plans/<文件名>.md`。两种执行方式：

1. **子代理驱动（推荐）** — 每个任务分派一个全新子代理，任务间审查，快速迭代

2. **逐步执行** — 在当前会话中逐步执行任务，批量执行带检查点

选择哪种方式？
```

## 危险信号 — 立即停下

| 想法 | 现实 |
|------|------|
| "计划太细了，浪费时间" | 粗糙的计划 = 执行时的猜测 = 返工 |
| "这个步骤太明显了，不用写" | 对你明显 ≠ 对执行者明显 |
| "类似任务 N，不重复了" | 执行者可能不按顺序读。重复代码 |
| "后面再补测试" | 测试是步骤的一部分，不是附加品 |
| "先写代码再写计划" | 这正是此 Skill 要防止的行为 |

## 常见借口反驳表

| 借口 | 现实 |
|------|------|
| "计划会过时" | 过时的计划比没有计划好。计划是起点，不是终点 |
| "敏捷不需要详细计划" | 敏捷 ≠ 没有计划。Sprint planning 就是计划 |
| "我边做边想更高效" | 边做边想 = 边做边返工。先想清楚 |
| "这个项目太小不需要计划" | 小项目的计划也小。5 分钟写计划省 1 小时返工 |
| "AI 足够聪明不需要这么细" | AI 在明确指令下表现最好。模糊指令 = 模糊结果 |

## 注意事项

1. **精确文件路径**：每个任务必须给出精确的文件路径
2. **完整代码**：每个涉及代码的步骤必须包含完整代码
3. **精确命令**：运行命令必须包含预期输出
4. **DRY + YAGNI + TDD**：这三个原则贯穿每个任务
5. **频繁提交**：每个任务完成后都要提交
