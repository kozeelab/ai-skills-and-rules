# 提示词自动优化器

## 功能说明

> **核心目标**：作为用户与 AI 之间的智能中间层，自动拦截用户的原始提示词，运用提示词工程最佳实践进行优化，然后将优化后的提示词发送给 AI，从而显著提升 AI 的响应质量。

本 Skill 的工作模式是**透明代理**——用户正常输入，Skill 在后台自动优化，用户无需学习提示词工程技巧即可获得高质量的 AI 响应。

### 核心能力

1. **结构化重组**：将模糊的自然语言转化为结构清晰的提示词
2. **上下文补全**：自动补充缺失的关键上下文信息
3. **约束明确化**：将隐含的需求转化为明确的约束条件
4. **角色设定注入**：根据任务类型自动注入最佳角色设定
5. **输出格式规范**：明确指定期望的输出格式和结构

---

## AI 执行流程

### Step 1：拦截并分析原始提示词

当用户发送消息时，AI 首先对原始提示词进行分析，识别以下要素：

```
分析维度：
├── 意图识别：用户想要完成什么任务？
├── 完整性评估：提示词是否包含足够的上下文？
├── 模糊度检测：是否存在歧义或模糊表达？
├── 约束缺失：是否缺少必要的约束条件？
└── 任务分类：属于哪类任务？（编码/写作/分析/创意/问答/其他）
```

**判断规则**：
- 如果原始提示词已经足够清晰和完整（评分 ≥ 8/10），则**跳过优化**，直接执行
- 如果原始提示词需要优化（评分 < 8/10），进入 Step 2

### Step 2：应用优化策略

根据任务分类和分析结果，选择并应用以下优化策略：

#### 策略 A：结构化重组

将松散的描述重组为结构化格式：

```
优化前：帮我写个登录页面，要好看点
优化后：
  任务：创建一个现代化的登录页面
  技术栈：[根据项目上下文自动推断]
  设计要求：
    - 采用现代简约设计风格
    - 响应式布局，适配移动端和桌面端
    - 包含用户名/邮箱和密码输入框
    - 包含"记住我"选项和"忘记密码"链接
    - 包含社交登录按钮（可选）
  交互要求：
    - 表单验证（前端实时校验）
    - 加载状态反馈
    - 错误提示友好
```

#### 策略 B：上下文补全

自动从当前环境中提取并注入上下文：

- **项目上下文**：技术栈、框架版本、项目结构
- **文件上下文**：当前打开的文件、最近编辑的文件
- **对话上下文**：之前的对话历史中的关键信息
- **环境上下文**：操作系统、开发工具、运行环境

#### 策略 C：约束明确化

将隐含需求转化为显式约束：

```
隐含需求 → 显式约束 示例：
"写个函数" → "编写一个函数，要求：参数类型明确、包含错误处理、有文档注释、包含单元测试示例"
"优化性能" → "优化性能，目标：响应时间降低 50%，内存占用不超过 XXX，需保持向后兼容"
```

#### 策略 D：角色设定注入

根据任务类型自动注入最佳角色：

| 任务类型 | 注入角色 |
|---------|---------|
| 编码任务 | 资深软件工程师，精通 [相关技术栈] |
| 架构设计 | 系统架构师，擅长 [相关领域] |
| 文案写作 | 专业文案策划，风格 [根据需求推断] |
| 数据分析 | 数据分析师，熟悉 [相关工具和方法] |
| Bug 修复 | 调试专家，擅长 [相关语言/框架] |
| 代码审查 | 代码审查专家，关注安全性、性能和可维护性 |

#### 策略 E：输出格式规范

根据任务类型指定输出格式：

- **编码任务**：要求包含代码、注释、使用示例
- **分析任务**：要求结构化输出（表格/列表/分点）
- **创意任务**：要求提供多个方案供选择
- **问答任务**：要求先给结论，再给解释

### Step 3：组装优化后的提示词

将各策略的优化结果组装为最终提示词，遵循以下模板：

```
[角色设定（如需要）]

## 任务描述
[重组后的核心需求]

## 上下文
[补全的上下文信息]

## 具体要求
[明确化的约束条件，按优先级排列]

## 输出要求
[期望的输出格式和结构]
```

### Step 4：透明执行

将优化后的提示词作为实际指令执行，**默认不向用户展示优化过程**。

**透明度控制**：
- **静默模式**（默认）：直接执行优化后的提示词，用户无感知
- **摘要模式**：在响应末尾附加一行简短说明，如 `💡 已自动优化提示词：补充了项目上下文和输出格式要求`
- **详细模式**：展示优化前后的对比和优化报告（需用户主动开启）

用户可通过以下指令切换模式：
- `#prompt-optimizer silent` — 静默模式
- `#prompt-optimizer summary` — 摘要模式
- `#prompt-optimizer verbose` — 详细模式
- `#prompt-optimizer off` — 关闭优化
- `#prompt-optimizer on` — 开启优化

### Step 5：持续学习与适应

在对话过程中，Skill 应持续观察：

1. **用户反馈信号**：如果用户对响应不满意并进行追问/纠正，记录优化失败点
2. **偏好学习**：识别用户的表达习惯和偏好，逐步调整优化策略
3. **上下文积累**：随着对话深入，积累更多上下文用于后续优化

## 优化规则库

### 通用优化规则

1. **具体化**：将抽象描述转化为具体要求
   - ❌ "写好一点" → ✅ "遵循 SOLID 原则，包含错误处理和文档注释"
2. **量化**：将模糊程度转化为可衡量指标
   - ❌ "快一点" → ✅ "响应时间 < 200ms"
3. **分步化**：将复杂任务拆解为可执行步骤
   - ❌ "帮我重构这个项目" → ✅ "分三步重构：1. 提取公共模块 2. 统一错误处理 3. 优化数据层"
4. **示例化**：提供输入输出示例
   - ❌ "转换数据格式" → ✅ "将 JSON 转换为 CSV，示例：{\"name\":\"test\"} → name\ntest"
5. **边界化**：明确限制和边界条件
   - ❌ "处理用户输入" → ✅ "处理用户输入，需校验：长度 1-100，仅允许字母数字，需防 XSS"

### 不优化的情况

以下情况应**跳过优化**，保持原始提示词：

1. 用户的提示词已经非常清晰和完整
2. 用户在进行简单的闲聊或问候
3. 用户明确要求关闭优化（`#prompt-optimizer off`）
4. 用户的提示词包含精确的技术指令（如 shell 命令、代码片段）
5. 用户正在进行多轮对话的追问/补充，上下文已充分

---

## 注意事项

1. **透明性原则**：优化过程默认对用户透明（静默模式），但用户随时可以查看优化详情
2. **保真原则**：优化不能改变用户的核心意图，只能增强和补充
3. **最小干预原则**：如果原始提示词已经足够好，不要过度优化
4. **上下文敏感**：优化策略应根据当前项目和对话上下文动态调整
5. **性能考虑**：优化过程应快速完成，不能显著增加响应延迟
6. **隐私保护**：优化过程中不应引入用户未提及的敏感信息
7. **可逆性**：用户可以随时关闭优化功能，回到原始交互模式
