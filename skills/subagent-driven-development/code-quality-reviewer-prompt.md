# 代码质量审查子代理 Prompt 模板

分派代码质量审查子代理时使用此模板。

**目的：** 验证实现是否构建良好（干净、经过测试、可维护）

**仅在规格合规审查通过后才分派。**

```
你正在对以下实现进行代码质量审查。

## 实现内容

[来自实现者报告的内容]

## 计划/需求

任务 N，来自 [计划文件路径]

## 变更范围

基准提交：[BASE_SHA]
当前提交：[HEAD_SHA]
描述：[任务摘要]

## 审查维度

### 1. 代码质量
- 代码是否干净、可读、可维护？
- 命名是否清晰准确？
- 是否遵循了项目的编码规范？
- 是否有代码异味（过长函数、过深嵌套、重复代码）？

### 2. 测试质量
- 测试是否覆盖了关键行为？
- 测试是否测试真实行为（而非 mock 行为）？
- 是否覆盖了边界情况和错误场景？
- 测试是否遵循了 TDD（如果要求）？

### 3. 架构质量
- 每个文件是否有一个明确的职责和定义良好的接口？
- 单元是否可以独立理解和测试？
- 实现是否遵循了计划中的文件结构？
- 这次变更是否创建了已经很大的新文件，或显著增长了现有文件？

### 4. 安全性
- 是否有注入风险（SQL、XSS、命令注入）？
- 错误信息是否暴露了内部细节？
- 输入是否经过校验？

### 5. 性能
- 是否有 N+1 查询？
- 是否有不必要的内存分配？
- 大数据量是否有分页？

## 报告格式

**优点：** [列出做得好的地方]

**问题：**
- 🔴 严重（Critical）：[阻塞性问题，必须修复]
- 🟡 重要（Important）：[应该修复再继续]
- 🔵 次要（Minor）：[可以后续修复]

**评估：** [通过 / 需要修复后重新审查]
```
