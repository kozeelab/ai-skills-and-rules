# 项目总结 Skill

## 功能说明

此Skill用于总结项目中的亮点和难点，生成一段**精炼且高质量**的、适合直接复制到简历上的项目经历。

> **核心目标**：在 **10-15行、不超过400字** 的篇幅内，让面试官在 **10秒扫读** 中抓住你的核心价值。

---

## 执行步骤

### Step 1：项目深度分析

- 扫描项目目录结构、技术栈和框架
- 分析核心业务逻辑和模块划分
- **重点识别**：你做的事情中，哪些是「别人做不了 / 不容易做到」的

### Step 2：价值提炼（核心环节）

按以下优先级筛选，**只保留最有价值的 5-7 条**，分配到职责/亮点/难点三个板块：

| 优先级 | 类型 | 适合放在 | 说明 |
|--------|------|----------|------|
| ⭐⭐⭐ | **从0到1** | 核心职责 | 独立设计/搭建的系统、架构、方案 |
| ⭐⭐⭐ | **量化显著** | 核心职责/亮点 | 性能提升>50%、成本降低>30%等有冲击力的数据 |
| ⭐⭐⭐ | **攻克难题** | 技术难点 | 别人搞不定的问题你搞定了（疑难Bug、架构瓶颈、技术选型困境） |
| ⭐⭐ | **技术深度** | 技术亮点 | 有难度的技术实现（并发、一致性、性能瓶颈等） |
| ⭐⭐ | **业务影响** | 核心职责 | 直接影响核心业务指标（收入、用户量、转化率） |
| ⭐ | **流程改进** | 核心职责 | 提升团队效率、建立规范、自动化 |

### Step 3：按模板输出

## 满分写作方法论

### 🎯 黄金公式

每条描述遵循 **「做了什么 → 怎么做的 → 效果如何」** 三段式，一句话写完：

| 层次 | 作用 | 示例片段 |
|------|------|----------|
| **做了什么** | 让面试官知道你的职责范围 | "设计并实现分布式任务调度系统" |
| **怎么做的** | 体现技术深度和方案能力 | "基于Redis+Lua实现分布式锁，结合延迟队列保证最终一致性" |
| **效果如何** | 用数据证明你的价值 | "日均调度50万+任务，故障率<0.01%" |

> ⚠️ **篇幅有限时**，「怎么做的」可以省略，但「做了什么」和「效果如何」**绝不能省**。

### 💎 难点写作公式

难点是面试中**最容易引发深度追问**的部分，写好了能直接拉开差距：

**公式：「问题有多难 → 你怎么解 → 效果多好」**

| 层次 | 作用 | 写法要点 |
|------|------|----------|
| **问题有多难** | 让面试官感受到挑战的份量 | 强调约束条件（时间紧、资源少、没有先例、影响范围大） |
| **你怎么解** | 体现你的思考深度和技术判断力 | 写关键决策点，而非流水账（为什么选A不选B） |
| **效果多好** | 用数据证明方案的有效性 | 对比解决前后的指标变化 |

> 🔑 **难点 vs 亮点的区别**：亮点侧重「我做了什么牛的事」，难点侧重「我解决了什么别人搞不定的问题」。
> 亮点体现能力广度，难点体现思考深度。两者结合才是满分。

### 🔑 差异化策略

让你的项目经历从千篇一律中脱颖而出：

1. **用具体技术名词替代泛泛描述**
   - ❌ "优化了数据库查询性能"
   - ✅ "通过索引重建+慢查询分析+读写分离，将核心接口P99从800ms降至120ms"

2. **用业务语言包装技术成果**
   - ❌ "减少了接口响应时间"
   - ✅ "首页加载耗时从3.2s降至0.8s，用户跳出率降低18%"

3. **体现决策能力而非执行能力**
   - ❌ "使用Redis做缓存"
   - ✅ "针对高频读低频写场景，设计二级缓存策略（本地缓存+Redis），缓存命中率达98%"

### 🚫 避坑清单（常见减分写法）

| 减分写法 | 问题 | 改进方向 |
|----------|------|----------|
| "负责XX模块的开发和维护" | 太泛，看不出做了什么 | 写具体做了什么功能/解决了什么问题 |
| "提升了系统性能" | 没有量化，无说服力 | 必须给出具体数字 |
| "使用了Spring Boot + MySQL + Redis" | 技术栈罗列，没有深度 | 说明为什么用、怎么用、解决了什么问题 |
| "与团队协作完成XX" | 看不出个人贡献 | 明确你个人负责的部分和成果 |
| 每条都写50+字的长句 | 面试官读不完 | 控制在30-45字，一句话说清楚 |
| "遇到了XX难题并解决了" | 没说难在哪、怎么解的 | 写清约束条件+关键决策+前后对比 |
| 难点和亮点内容重复 | 浪费篇幅，信息密度低 | 亮点写方案能力，难点写攻坚能力，互补不重叠 |

## 量化指标参考

| 类型 | 高分示例 |
|------|----------|
| 性能 | P99延迟从Xms降至Yms、QPS从X提升至Y、内存占用减少X% |
| 效率 | 开发周期缩短X%、部署频率从周级提升至日级、人工操作减少X% |
| 质量 | 线上故障率降低X%、代码覆盖率提升至X%、可用性达99.X% |
| 业务 | DAU增长X%、转化率提升X个百分点、年节省成本X万 |
| 规模 | 日均处理X万请求、支撑X万用户并发、管理X+微服务 |
| 难度 | 无现成方案自研实现、在X约束下完成Y、排查耗时X天的疑难问题 |

## 岗位适配策略

根据目标岗位调整侧重点：

| 目标岗位 | 核心职责侧重 | 技术亮点侧重 | 技术难点侧重 |
|----------|-------------|-------------|-------------|
| **初/中级开发** | 独立完成的功能模块、代码质量 | 解决的具体技术问题、性能优化 | 排查定位的疑难Bug、独立攻克的技术障碍 |
| **高级开发** | 核心模块设计、技术方案选型 | 架构设计、复杂问题解决、技术深度 | 高并发/一致性等复杂场景的方案权衡与落地 |
| **架构师** | 系统架构设计、技术决策 | 高可用/高并发方案、技术演进路线 | 系统级瓶颈突破、跨团队技术难题协调解决 |
| **技术管理** | 团队搭建、技术规划、跨团队协作 | 技术体系建设、效能提升、人才培养 | 资源约束下的技术决策、遗留系统改造 |

> AI 在生成时应**主动询问**用户的目标岗位级别，以便调整内容侧重。

## 示例输出

### 示例1：后端开发（高级）

```
**智能营销平台** | Go, gRPC, Kafka, ClickHouse, K8s | 8人 | 12个月
面向电商场景的实时营销决策系统，支撑日均2000万+事件处理，驱动千万级用户的精准触达。

核心职责：
1. 主导设计事件驱动架构，基于Kafka实现毫秒级实时数据管道，日均处理2000万+事件，端到端延迟<200ms
2. 搭建规则引擎与A/B实验平台，支持运营零代码配置营销策略，策略上线周期从3天缩短至2小时
3. 设计用户画像宽表方案（ClickHouse），将圈人查询从分钟级优化至秒级，支撑千万级用户实时圈选

技术亮点：
1. 针对高并发写入场景，设计批量聚合+异步刷盘策略，ClickHouse写入吞吐提升5倍，存储成本降低40%
2. 实现基于滑动窗口的限流熔断机制，在下游服务抖动时保障核心链路可用性达99.99%

技术难点：
1. 大促期间Kafka消费积压达千万级，通过动态分区扩容+消费者组弹性伸缩+背压感知降级，30分钟内消化积压且零数据丢失
2. 用户画像宽表字段达500+列导致查询超时，调研对比多种方案后采用列式存储+物化视图预聚合，查询耗时从47s降至1.2s
```

### 示例2：全栈开发（中级）

```
**企业协同办公平台** | React, Node.js, TypeScript, PostgreSQL, WebSocket | 5人 | 8个月
面向中小企业的一站式协同办公系统，覆盖项目管理、文档协作、即时通讯，服务500+企业、2万+活跃用户。

核心职责：
1. 独立开发实时文档协作模块，基于OT算法实现多人同时编辑，支持50人同时在线编辑零冲突
2. 设计并实现RBAC权限体系，支持组织架构级联授权，覆盖200+细粒度权限点
3. 搭建WebSocket长连接网关，实现消息实时推送，在线用户消息到达率99.5%

技术亮点：
1. 针对文档协作的并发冲突问题，实现基于操作变换（OT）的冲突解决算法，编辑延迟<100ms
2. 通过虚拟列表+增量渲染优化万级数据列表，首屏渲染时间从2.8s降至0.6s，内存占用减少65%

技术难点：
1. 多人协作时OT算法在弱网环境下出现操作乱序，设计版本向量+操作缓冲队列机制，弱网（300ms延迟）下编辑一致性从87%提升至99.9%
```
