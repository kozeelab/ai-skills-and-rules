# Skill 质量守护者

## 功能说明

> **核心目标**：作为整个 Skill 体系的**质量守护者**，持续学习最新的 AI 领域知识（Prompt Engineering、Agent 设计模式、LLM 最佳实践等），基于前沿知识对所有 Skill 进行多维度质量审查，自动发现不足并完善，确保每个 Skill 都是高质量的、与时俱进的。

本 Skill 的工作模式是**知识引擎 + 质量审计师**——持续吸收最新 AI 知识，将知识转化为审查标准，对新创建和已有的 Skill 进行全方位质量审查和自动完善。

### 核心能力

1. **知识自学习**：持续学习最新的 AI/LLM/Agent/Prompt Engineering 知识，构建前沿知识库
2. **质量审查**：基于多维度质量模型对 Skill 进行全面审查
3. **自动完善**：发现质量问题后自动修复和增强 Skill 内容
4. **一致性保障**：确保所有 Skill 遵循统一的设计模式和质量标准
5. **持续进化**：随着 AI 知识更新，自动推动已有 Skill 升级

## AI 知识自学习体系

### 知识来源与分类

> AI 应持续关注以下知识领域，并将最新知识融入 Skill 质量审查标准中。

#### 核心知识领域

| 知识领域 | 关键主题 | 对 Skill 质量的影响 |
|---------|---------|-------------------|
| **Prompt Engineering** | 结构化提示词、思维链（CoT）、少样本学习、角色设定、约束工程 | Skill 的指令设计是否运用了最新的提示词技术 |
| **Agent 设计模式** | ReAct、Plan-and-Execute、Reflection、Tool Use、Multi-Agent | Skill 的执行流程是否采用了最优的 Agent 模式 |
| **LLM 能力边界** | 上下文窗口管理、幻觉防护、推理能力、代码生成能力 | Skill 是否合理利用了 LLM 的能力并规避了局限 |
| **AI 安全与对齐** | 提示注入防护、输出安全、偏见检测、可控性 | Skill 是否考虑了安全性和可控性 |
| **人机协作模式** | 人在回路（HITL）、渐进式自动化、透明度、可解释性 | Skill 的人机交互设计是否合理 |
| **AI 工具链** | MCP 协议、Function Calling、RAG、向量检索 | Skill 是否充分利用了可用的工具和能力 |

#### 知识更新策略

```
知识更新触发时机：
├── 1. 每次对话开始时 — 基于 AI 自身训练知识进行知识刷新
├── 2. 创建新 Skill 时 — 检查是否有更新的设计模式可以应用
├── 3. 用户触发全量审查时 — 对所有 Skill 应用最新知识
├── 4. 发现 Skill 执行效果不佳时 — 分析是否有更好的实现方式
└── 5. AI 知识库更新时 — 自动评估对现有 Skill 的影响
```

### 知识转化为审查标准

AI 应将学习到的知识转化为可执行的审查标准：

```
知识学习 → 提取关键原则 → 转化为检查项 → 纳入质量模型
                                              ↓
                                    应用于 Skill 审查
```

**示例**：

| 学到的知识 | 提取的原则 | 转化的检查项 |
|-----------|-----------|------------|
| CoT（思维链）可以提升复杂推理质量 | 复杂任务应引导 AI 分步思考 | Skill 的复杂执行流程是否包含分步推理指引？ |
| 结构化输出比自由文本更可靠 | 输出应有明确的格式约束 | Skill 是否定义了结构化的输出模板？ |
| 角色设定可以显著影响输出质量 | 任务相关的角色设定很重要 | Skill 是否为 AI 设定了合适的角色？ |
| Few-shot 示例比纯指令更有效 | 关键行为应提供示例 | Skill 是否包含充分的示例场景？ |
| 过长的提示词会导致注意力稀释 | 指令应精炼聚焦 | Skill 是否存在冗余或重复的指令？ |
| Reflection 模式可以提升输出质量 | 重要输出应有自我审查环节 | Skill 是否包含输出质量自检机制？ |

## Skill 质量模型（多维度）

### 质量维度总览

```
Skill 质量模型
├── 1. 结构完整性（Structural Completeness）
├── 2. 指令有效性（Instruction Effectiveness）
├── 3. 执行流程合理性（Flow Rationality）
├── 4. 知识前沿性（Knowledge Currency）
├── 5. 人机交互设计（Human-AI Interaction）
├── 6. 鲁棒性与容错（Robustness）
├── 7. 可维护性与可扩展性（Maintainability）
├── 8. 协作兼容性（Collaboration Compatibility）
└── 9. 示例充分性（Example Sufficiency）
```

### 维度 1：结构完整性

> 检查 Skill 文件的结构是否完整、规范。

```
检查项：
├── [ ] YAML Front Matter 是否完整？
│   ├── name / displayName / description / version
│   ├── author（name + url）
│   ├── category / tags
│   ├── compatibility（platforms + languages）
│   ├── input（description + required + optional）
│   └── output（description + artifacts）
├── [ ] 功能说明是否清晰？
│   ├── 核心目标（一句话概括）
│   ├── 工作模式描述（用类比说明）
│   └── 核心能力列表（3-6 项）
├── [ ] 执行流程是否完整？
│   ├── 每个 Step 有清晰的标题和描述
│   ├── Step 之间有明确的流转关系
│   └── 包含流程图或伪代码
├── [ ] 是否有用户控制指令？（自动化 Skill 必须有）
├── [ ] 是否有示例场景？（至少 2 个）
├── [ ] 是否有注意事项？（至少 3 条）
├── [ ] 是否有与其他 Skill 的协作说明？
└── [ ] 是否有版本号且遵循语义化版本？
```

**严重级别**：

| 缺失项 | 级别 |
|--------|------|
| YAML Front Matter 不完整 | 🔴 阻断 |
| 缺少功能说明或执行流程 | 🔴 阻断 |
| 缺少示例场景 | 🟡 警告 |
| 缺少注意事项 | 🟡 警告 |
| 缺少协作说明 | 🔵 建议 |

### 维度 2：指令有效性

> 检查 Skill 中的指令是否能被 AI 有效执行。

```
检查项：
├── [ ] 指令是否具体明确？（无歧义，AI 可直接执行）
├── [ ] 是否运用了思维链（CoT）引导？（复杂任务分步推理）
├── [ ] 是否有角色设定？（明确 AI 在执行此 Skill 时的角色）
├── [ ] 约束条件是否清晰？（什么该做、什么不该做）
├── [ ] 是否有优先级标注？（多个规则冲突时如何选择）
├── [ ] 是否避免了过度指令？（指令过多导致注意力稀释）
├── [ ] 关键术语是否有定义？（避免理解偏差）
├── [ ] 是否使用了结构化格式？（表格/列表/代码块而非大段文字）
├── [ ] 触发条件是否明确？（什么时候激活、什么时候跳过）
└── [ ] 是否有「不做什么」的负面约束？（防止过度执行）
```

**常见问题与修复**：

| 问题 | 修复方式 |
|------|---------|
| 指令过于抽象 | 添加具体的检查项列表和判断标准 |
| 缺少分步引导 | 将复杂流程拆解为 Step 1/2/3... |
| 没有角色设定 | 在功能说明中明确 AI 的角色定位 |
| 约束不清晰 | 添加「注意事项」章节，列出边界条件 |
| 指令冗余重复 | 合并重复内容，提取公共部分 |
| 缺少触发条件 | 添加明确的激活/跳过判断规则 |

### 维度 3：执行流程合理性

> 检查 Skill 的执行流程是否设计合理。

```
检查项：
├── [ ] 流程是否有明确的起点和终点？
├── [ ] 每个步骤的输入输出是否明确？
├── [ ] 步骤之间的依赖关系是否正确？（无循环依赖）
├── [ ] 是否有异常处理路径？（某步骤失败时怎么办）
├── [ ] 是否有最大执行轮次限制？（防止无限循环）
├── [ ] 是否采用了合适的 Agent 模式？
│   ├── 简单任务：直接执行模式
│   ├── 复杂任务：Plan-and-Execute 模式
│   ├── 需要自我改进：Reflection 模式
│   └── 需要持续监控：Observer 模式
├── [ ] 是否有渐进式执行策略？（先核心后细节）
├── [ ] 是否考虑了性能影响？（不应显著增加响应延迟）
└── [ ] 是否有状态管理机制？（跨轮次的状态如何保持）
```

### 维度 4：知识前沿性

> 检查 Skill 是否运用了最新的 AI 知识和最佳实践。

```
检查项：
├── [ ] 是否运用了最新的 Prompt Engineering 技术？
│   ├── 结构化提示词（而非纯自然语言指令）
│   ├── 思维链引导（复杂推理场景）
│   ├── 少样本示例（关键行为示范）
│   ├── 约束工程（明确边界和限制）
│   └── 输出格式控制（结构化输出模板）
├── [ ] 是否采用了合适的 Agent 设计模式？
│   ├── ReAct（推理 + 行动交替）
│   ├── Reflection（自我审查和改进）
│   ├── Plan-and-Execute（先规划后执行）
│   └── Tool Use（合理利用工具能力）
├── [ ] 是否考虑了 LLM 的能力边界？
│   ├── 上下文窗口限制（长文档分段处理）
│   ├── 幻觉防护（要求引用来源/验证事实）
│   ├── 推理局限（复杂逻辑分步验证）
│   └── 一致性保障（关键输出交叉验证）
├── [ ] 是否有过时的设计模式需要升级？
└── [ ] 是否充分利用了可用的工具和能力？
```

### 维度 5：人机交互设计

> 检查 Skill 的人机交互是否合理。

```
检查项：
├── [ ] 透明度是否合适？（用户能了解 Skill 在做什么）
├── [ ] 是否有用户控制机制？（开关/模式切换/参数调整）
├── [ ] 输出格式是否友好？（结构化、易读、有层次）
├── [ ] 是否有进度反馈？（长时间执行时告知用户进度）
├── [ ] 是否遵循了「最小惊讶原则」？（行为符合用户预期）
├── [ ] 是否有优雅降级？（功能受限时仍能提供基本服务）
├── [ ] 错误信息是否友好？（告知用户问题和解决方案）
├── [ ] 是否避免了信息过载？（默认精简，详细信息按需展示）
└── [ ] 是否有帮助/使用指南？（用户控制指令表）
```

### 维度 6：鲁棒性与容错

> 检查 Skill 在异常情况下的表现。

```
检查项：
├── [ ] 输入缺失时是否有合理的默认行为？
├── [ ] 依赖的文件不存在时是否有降级策略？
├── [ ] 执行过程中出错时是否有恢复机制？
├── [ ] 是否有防止无限循环的保护？（最大轮次/超时）
├── [ ] 多个 Skill 同时运行时是否有冲突处理？
├── [ ] 是否考虑了边界情况？（空输入/超大输入/特殊字符）
├── [ ] 是否有幂等性保障？（重复执行不产生副作用）
└── [ ] 是否有回滚机制？（修改出错时可以撤销）
```

### 维度 7：可维护性与可扩展性

> 检查 Skill 是否易于维护和扩展。

```
检查项：
├── [ ] 文档是否清晰完整？（新人能快速理解）
├── [ ] 是否有版本管理？（语义化版本号）
├── [ ] 规则是否参数化？（可通过配置调整而非修改文档）
├── [ ] 是否有扩展点？（新增功能不需要大幅重写）
├── [ ] 是否遵循了 DRY 原则？（无重复内容）
├── [ ] 是否有明确的变更影响范围？（修改某处不会意外影响其他部分）
├── [ ] 是否与其他 Skill 松耦合？（可独立使用）
└── [ ] 是否有废弃/迁移策略？（功能变更时的平滑过渡）
```

### 维度 8：协作兼容性

> 检查 Skill 与其他 Skill 的协作是否顺畅。

```
检查项：
├── [ ] 是否在 skills/index.md 中正确注册？
├── [ ] 与其他 Skill 的协作关系是否明确？
├── [ ] 是否有输入输出接口定义？（便于其他 Skill 调用）
├── [ ] 执行顺序是否与其他 Skill 兼容？
├── [ ] 是否会与其他 Skill 产生冲突？（如同时修改同一文件）
├── [ ] 是否正确标注了自动化属性？（便于 skill-auto-activator 识别）
└── [ ] 是否遵循了 Skill 体系的统一设计模式？
```

### 维度 9：示例充分性

> 检查 Skill 的示例是否充分且有代表性。

```
检查项：
├── [ ] 是否有至少 2 个示例场景？
├── [ ] 示例是否覆盖了核心功能？（不只是简单场景）
├── [ ] 是否有正常流程的示例？（Happy Path）
├── [ ] 是否有异常处理的示例？（Error Path）
├── [ ] 是否有边界情况的示例？（Edge Case）
├── [ ] 示例是否具体且可执行？（不是抽象描述）
├── [ ] 示例的输入输出是否完整？
└── [ ] 示例是否展示了 Skill 的核心价值？
```

## AI 执行流程

### Step 1：知识刷新（每次审查前执行）

```
🧠 AI 知识刷新
━━━━━━━━━━━━━━
1. 回顾最新的 Prompt Engineering 技术
   ├── 结构化提示词设计
   ├── 思维链（CoT）/ 思维树（ToT）
   ├── 少样本学习（Few-shot）
   ├── 自我一致性（Self-Consistency）
   └── 约束工程与输出控制
2. 回顾最新的 Agent 设计模式
   ├── ReAct / Plan-and-Execute / Reflection
   ├── Multi-Agent 协作
   ├── Tool Use 最佳实践
   └── 状态管理与记忆机制
3. 回顾 LLM 能力边界与最佳实践
   ├── 上下文窗口优化
   ├── 幻觉防护策略
   ├── 推理能力增强
   └── 安全与对齐
4. 整理知识更新清单
   └── 列出可应用于 Skill 改进的知识点
━━━━━━━━━━━━━━
```

### Step 2：加载审查目标

根据审查模式确定目标 Skill：

```
审查模式判断：
├── new  → 仅审查本次对话中新创建的 Skill
├── all  → 读取 skills/index.md，审查所有 Skill
└── target → 审查用户指定的 Skill 文件
```

**加载流程**：
1. 读取 `skills/index.md` 获取 Skill 列表
2. 读取目标 Skill 文件的完整内容
3. 读取相关的协作 Skill 文件（了解协作关系）
4. 构建审查上下文

### Step 3：执行多维度质量审查

对每个目标 Skill，按 9 个维度逐项审查：

```
┌─────────────────────────────────────────────────┐
│              Skill 质量审查流程                    │
│                                                 │
│  目标 Skill                                      │
│      ↓                                          │
│  维度 1：结构完整性 ──→ 发现问题记录              │
│      ↓                                          │
│  维度 2：指令有效性 ──→ 发现问题记录              │
│      ↓                                          │
│  维度 3：执行流程合理性 ──→ 发现问题记录          │
│      ↓                                          │
│  维度 4：知识前沿性 ──→ 发现问题记录              │
│      ↓                                          │
│  维度 5：人机交互设计 ──→ 发现问题记录            │
│      ↓                                          │
│  维度 6：鲁棒性与容错 ──→ 发现问题记录            │
│      ↓                                          │
│  维度 7：可维护性与可扩展性 ──→ 发现问题记录      │
│      ↓                                          │
│  维度 8：协作兼容性 ──→ 发现问题记录              │
│      ↓                                          │

│      ↓                                          │
│  汇总 → 生成审查报告 → 进入修复流程              │
└─────────────────────────────────────────────────┘
```

**审查评分标准**：

| 维度 | 权重 | 满分 | 及格线 |
|------|------|------|--------|
| 结构完整性 | 15% | 10 | 8 |
| 指令有效性 | 20% | 10 | 7 |
| 执行流程合理性 | 15% | 10 | 7 |
| 知识前沿性 | 10% | 10 | 6 |
| 人机交互设计 | 10% | 10 | 7 |
| 鲁棒性与容错 | 10% | 10 | 6 |

### Step 5：自动完善循环

当审查发现问题时，进入自动完善循环：

```
┌─────────────────────────────────────────────────┐
│              自动完善循环                          │
│                                                 │
│  审查发现问题                                    │
│      ↓                                          │
│  按严重级别排序（🔴 优先）                        │
│      ↓                                          │
│  逐项自动修复                                    │
│  ├── 结构缺失 → 补充缺失的章节/字段              │
│  ├── 指令模糊 → 重写为具体明确的指令              │
│  ├── 流程缺陷 → 重新设计执行流程                  │
│  ├── 知识过时 → 应用最新知识升级                  │
│  ├── 交互不佳 → 优化输出格式和控制指令            │
│  ├── 容错不足 → 添加异常处理和降级策略            │
│  ├── 示例不足 → 补充示例场景                      │
│  └── 协作缺失 → 添加协作关系说明                  │
│      ↓                                          │
│  重新审查修复后的 Skill                           │
│      ↓                                          │
│  ┌─ 全部通过（≥ 良好） → 输出完善后的 Skill ✅    │
│  │                                              │
│  ├─ 仍有问题且轮次 < 3                           │
│  │   → 继续修复（下一轮）                        │
│  │                                              │
│  └─ 达到最大轮次仍有问题                          │
│      → 输出 Skill + 标注剩余问题 ⚠️              │
└─────────────────────────────────────────────────┘
```

**完善策略详解**：

| 问题类型 | 自动完善方式 |
|---------|------------|
| YAML Front Matter 不完整 | 根据 Skill 内容自动补充缺失字段 |
| 缺少功能说明 | 根据执行流程反推功能说明 |
| 指令过于抽象 | 添加具体的检查项列表、判断标准和阈值 |
| 缺少思维链引导 | 将复杂流程拆解为分步推理 |
| 缺少角色设定 | 根据 Skill 类型注入合适的角色描述 |
| 执行流程缺少异常路径 | 添加错误处理和降级策略 |
| 缺少防循环保护 | 添加最大轮次限制 |
| 未运用最新 Prompt 技术 | 应用 CoT/Few-shot/约束工程等技术重写指令 |
| 未采用最优 Agent 模式 | 根据任务特点推荐并应用合适的 Agent 模式 |
| 输出格式不友好 | 重新设计输出模板，使用表格/进度条/emoji |
| 缺少用户控制指令 | 添加 on/off/status 等控制指令 |
| 示例不足 | 补充 Happy Path / Error Path / Edge Case 示例 |
| 缺少协作说明 | 分析与其他 Skill 的关系并添加协作说明 |
| 与其他 Skill 格式不一致 | 统一为体系内的标准格式 |

## 注意事项

1. **业务逻辑不可变原则**：自动完善只改善 Skill 的质量（结构、指令、示例等），**绝不改变 Skill 的核心功能和业务逻辑**
2. **知识审慎应用原则**：不是所有最新知识都适合所有 Skill，应根据 Skill 的具体场景选择性应用
3. **最小修改原则**：完善时只修改必要的部分，不做无关的重写
4. **一致性优先原则**：完善后的 Skill 必须与体系内其他 Skill 保持风格一致
5. **防过度优化原则**：简单的 Skill 不需要复杂的设计模式，避免过度工程化
6. **用户确认原则**：对于大幅度的 Skill 重写，应先向用户展示改进方案并获得确认
7. **版本管理原则**：每次完善后应更新 Skill 的版本号（小改进 patch，大改进 minor）
8. **知识可追溯原则**：每次应用的最新知识点都应在审查报告中列出，便于用户了解改进依据
9. **渐进式改进原则**：对于评分较低的 Skill，优先修复 🔴 阻断项，再逐步提升其他维度，避免一次性大幅重写
10. **自我审查原则**：本 Skill 自身也应定期接受质量审查（可由用户手动触发 `#skill-quality check skill-quality-guardian`）
