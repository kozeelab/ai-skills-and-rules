# 错误处理规范

> **强制**：所有错误处理必须遵守本文件中的规范

## 1. 错误分类

### 1.1 业务错误
- 由业务逻辑产生的错误
- 应使用自定义错误码
- 错误信息应清晰明确

### 1.2 系统错误
- 由系统或外部依赖产生的错误
- 应包装原始错误
- 避免直接返回系统错误给前端

### 1.3 HTTP 错误
- 对应 HTTP 状态码的错误
- 如 400、401、403、404、500 等

## 2. 错误定义

### 2.1 错误码管理
- 使用专门的错误码管理机制
- 错误码应唯一且有意义
- 错误码应包含错误类型和具体错误

## 3. 错误传递

### 3.1 错误返回
- 错误应作为函数的返回值
- 必须处理不得忽略
- 采用独立错误流，禁止与其他变量组合判断

### 3.2 错误包装
- 业务层应包装底层错误
- 保留原始错误信息
- 添加适当的上下文信息

### 3.3 错误检查
- 使用适当的方法检查特定错误
- 正确处理不同类型的错误

## 4. 错误处理

### 4.1 统一响应
- **必须**使用统一的错误处理机制
- **严禁**直接返回错误给前端
- 统一错误响应格式

### 4.2 控制器层错误处理
- 控制器层应捕获所有错误
- 使用统一的错误处理函数
- 避免在控制器层处理业务逻辑错误

## 5. 错误日志

### 5.1 日志记录
- 错误必须立即记录
- 日志应包含错误上下文
- 避免日志中包含敏感信息

### 5.2 日志级别
- 错误：使用错误级别日志
- 警告：使用警告级别日志
- 信息：使用信息级别日志
- 调试：使用调试级别日志

## 6. 错误处理最佳实践

### 6.1 分层错误处理
- 数据访问层：返回原始错误
- 业务逻辑层：包装错误，添加业务上下文
- 控制器层：统一处理错误，返回标准化响应

### 6.2 错误信息设计
- 错误信息应清晰明确
- 对用户友好，不暴露内部实现细节
- 包含足够的信息便于调试

### 6.3 错误码使用
- 统一错误码管理
- 错误码与 HTTP 状态码对应
- 业务错误使用自定义错误码
