# API 设计规范

> **强制**：所有 API 设计必须遵守本文件中的规范

## 1. RESTful API 基本原则

### 1.1 URL 设计
- URL 使用名词复数，HTTP Method 区分语义
- 路径段小写 + 中划线
- 版本前缀 `/api/v1/`

```
// ✅ 正确
GET /api/v1/users           // 获取用户列表
POST /api/v1/users          // 创建用户
GET /api/v1/users/{id}      // 获取单个用户
PUT /api/v1/users/{id}      // 更新用户
DELETE /api/v1/users/{id}   // 删除用户

// ❌ 错误
GET /api/v1/getUser         // 使用动词
GET /api/v1/user/{id}       // 单数名词
GET /api/v1/Users/{id}      // 大小写混合
```

### 1.2 HTTP Method 语义
- POST：创建资源
- GET：查询资源
- PUT：更新资源（完整更新）
- PATCH：更新资源（部分更新）
- DELETE：删除资源

### 1.3 状态码使用
- 200 OK：成功（GET、PUT、PATCH）
- 201 Created：创建成功（POST）
- 204 No Content：删除成功（DELETE）
- 400 Bad Request：请求参数错误
- 401 Unauthorized：未授权
- 403 Forbidden：禁止访问
- 404 Not Found：资源不存在
- 409 Conflict：资源冲突
- 500 Internal Server Error：服务器内部错误

## 2. 路由注册

### 2.1 路由集中注册
- 所有路由应集中注册，便于管理和维护
- 按资源类型分组
- 鉴权通过中间件挂载

### 2.2 路由分组
- 按业务模块分组
- 统一应用中间件

## 3. 请求处理

### 3.1 参数绑定
- JSON Body 应使用框架提供的参数绑定功能
- Query 参数应使用专门的绑定方法
- 路径参数应通过框架提供的方法获取
- 禁止使用不安全的绑定方式

### 3.2 参数校验
- 请求参数必须进行校验
- 绑定失败应立即返回错误并终止处理
- 路径参数获取后必须校验非空

### 3.3 分页查询
- 列表接口应支持分页
- 分页参数应包括页码和每页大小
- 默认页码为1，每页大小为20，最大不超过100
- 返回结果应包含总条数、当前页码、每页大小和数据列表
- 排序字段应通过白名单校验

## 4. 响应处理

### 4.1 统一响应格式
- **必须**使用统一的响应函数处理成功和失败情况
- **严禁**直接拼装响应体
- 业务错误应通过统一的错误处理机制处理

### 4.2 响应结构
- 成功响应：`{"code": 0, "message": "success", "data": {...}}`
- 错误响应：`{"code": 错误码, "message": "错误信息"}`

## 5. 数据传输对象（DTO）设计规范

### 5.1 DTO 设计原则
- 创建DTO：值类型，必填字段应标记
- 更新DTO：指针类型（区分不传和清空）
- 请求/响应DTO分离，统一管理
- 每个字段必须有注释（含义+枚举值+默认值+约束）

## 6. API 文档

### 6.1 文档要求
- 新增 API 必须包含完整的文档
- 文档应包括接口描述、参数说明、返回值说明等
- 接口变更时同步更新文档

## 7. 中间件规范

### 7.1 中间件职责
- 中间件职责应单一
- 上下文信息应通过框架提供的机制传递
- 错误必须终止请求链

## 8. 安全规范

### 8.1 输入校验
- 用户输入必须校验长度/格式/范围
- 防止 SQL 注入、XSS 攻击等安全问题

### 8.2 敏感信息
- 敏感信息（密码/Token/Secret Key）禁止出现在日志和错误响应中
- 密钥禁止硬编码，通过配置文件或环境变量管理

### 8.3 权限控制
- 基于角色的访问控制
- 接口级权限检查
