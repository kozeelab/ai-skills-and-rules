# AI 协作开发规范

> **AI 必须严格遵守本文件中所有标注「强制」的规则，不得以任何理由跳过。**

---

## 一、强制执行协议

> ⚠️ **任何涉及代码修改的任务，必须先完成本协议，再动手写代码。没有"太简单不需要走流程"的例外。**

### 流程总览

```
收到任务 → Phase 1: 启动检查 → Phase 2: 执行 → Phase 3: 收尾检查
              ↓ 必须输出           ↓ 写代码        ↓ 必须输出
           「📋 任务启动清单」                    「✅ 任务完成清单」
```

### Phase 1：任务启动清单（写代码前必须输出）

```
📋 任务启动清单
━━━━━━━━━━━━━━
• 任务类型：[新增功能 / 修改接口 / Bug 修复 / 重构 / 其他]
• 需读取的参考文件：[列出需要先读取的文件]
• 涉及的规范要点：[从本文档中列出本次相关的强制项]
• 预计变更文件：[列出将要修改/新增的文件]
```

### Phase 3：任务完成清单（写完代码后必须输出）

```
✅ 任务完成清单
━━━━━━━━━━━━━━
1. 质量自检（逐项打勾，详见第四章）：[通过 / 列出未通过项]
2. 索引更新：[已更新 .ai/index.md / 无需更新]
3. 建议 commit message：`type(scope): description`
```

## 二、架构约束（强制）

### 2.1 分层与依赖方向

- 项目应遵循清晰的分层架构，各层职责单一
- 依赖方向必须单向（上层 → 下层），**严禁逆向依赖**
- 具体分层结构以项目实际架构为准，AI 应通过阅读项目代码识别

### 2.2 强制规范

- 统一响应格式：所有 API 必须通过统一的响应工具函数返回，**禁止**直接拼装响应体
- 所有导出的结构体/DTO 字段必须有注释
- 新增 API 必须有完整的 Swagger / OpenAPI 文档注释
- 错误处理必须分层传递，禁止吞掉错误
- 详细的 API 设计规范见 [api-design.md](./api-design.md)，错误处理规范见 [error-handling.md](./error-handling.md)，数据库规范见 [database.md](./database.md)

## 三、上下文管理（强制）

> AI 在动手写代码之前，必须先建立充分的上下文。

### 3.1 索引优先原则

任何涉及"定位文件"的操作，**必须优先读取 `.ai/index.md`**，禁止盲目搜索。适用场景：修改已有功能、多模块改动、Bug 追踪、新增 API 确认路由冲突、新增表确认关联关系。

### 3.2 Skill 按需加载

当用户的需求**不是普通编码任务**，而是涉及特定工作流（如简历项目总结、Git 多环境配置等）时，**先读取 `skills/index.md` 查找匹配的 Skill**，再按 Skill 定义的流程执行。普通编码任务无需读取 Skill 索引。

### 3.2.1 工作流编排（强制）

> ⚠️ 当用户的需求触发了工作流 Skill 时，AI **必须**按工作流链路顺序执行，不得跳过任何环节。

**完整开发工作流**（新功能/新项目）：
```
brainstorming → writing-plans → subagent-driven-development → test-driven-development → code-review → verification-before-completion
```

**Bug 修复工作流**：
```
systematic-debugging → test-driven-development → verification-before-completion
```

**触发规则**：

| 用户意图 | 触发的工作流 | 起始 Skill |
|---------|------------|-----------|
| 新功能/新项目 | 完整开发工作流 | brainstorming |
| Bug 修复 | Bug 修复工作流 | systematic-debugging |
| 简单代码修改 | 无需工作流 | 直接执行（遵守 rules） |
| 重构 | 完整开发工作流（简化版） | brainstorming |

**铁律**：
- 工作流中的每个 Skill 都是**门禁**，必须通过才能进入下一步
- brainstorming 未完成设计 → 禁止写计划
- writing-plans 未完成计划 → 禁止写代码
- test-driven-development 未通过 → 禁止声称完成
- verification-before-completion 未验证 → 禁止提交

### 3.3 Skill 调用通知（强制）

> AI 在使用 Skill 时，必须向用户明确告知 Skill 的运行状态，确保透明可感知。

#### 3.3.1 主动调用 Skill 时

当 AI 开始使用某个 Skill 执行任务时，**必须在执行前打印通知**，使用 Skill 的**中文名称**：

```
🚀 开始调用「{Skill 中文名}」
```

示例：
```
🚀 开始调用「提示词自动优化器」
🚀 开始调用「代码自动审查与修复器」
🚀 开始调用「Skill 质量守护者」
```

#### 3.3.2 后台自动 Skill 状态通知

对于在后台自动运行的 Skill（即具有 `#xxx on/off` 控制指令、持续工作模式的 Skill），**每轮用户对话开始时**，必须首先打印当前后台工作的 Skill 列表，使用 Skill 的**中文名称**，格式如下：

```
🔄 {N} 个 Skill 正在后台默默工作：
1. {Skill 中文名 A}
2. {Skill 中文名 B}
3. {Skill 中文名 C}
...
```

示例：
```
🔄 5 个 Skill 正在后台默默工作：
1. Skill 自动激活守护器
2. AI 不足捕捉与规则生成器
3. 提示词自动优化器
4. 代码自动审查与修复器
5. Skill 质量守护者
```

**注意事项**：
- 仅打印当前处于**开启状态**的后台 Skill，已被用户关闭的不打印
- 如果所有后台 Skill 均已关闭，则不打印此通知
- Skill 中文名称取自各 Skill 文件 YAML Front Matter 中的 `displayName` 字段

### 3.4 场景 → 主动读取参考文件

| 场景                  | 必须主动读取的文件                                        |
| --------------------- | --------------------------------------------------------- |
| **新增 API**          | 路由注册文件、同类 Handler/Service/Model、DTO/Entity 定义 |
| **修改现有接口**      | 目标 Handler、对应 Service/Model、相关 DTO                |
| **新增数据库表/字段** | Entity/Model 定义、DTO 文件、数据库初始化/迁移文件        |
| **Bug 修复**          | 出问题的文件、调用方和被调用方                            |
| **重构**              | 所有受影响的文件、调用方文件                              |

### 3.5 行为约束

- **禁止猜测**：缺少关键参考文件时，禁止凭空猜测字段名/类型/结构体定义/函数存在性/表结构
- **禁止过度读取**：不要一次性读取超过 8 个文件，分阶段按需读取
- **必须参考范本**：生成新代码时，必须找到同类实体的现有实现作为范本，禁止从零凭空生成
- **先读后改**：修改已有功能时，必须先读取目标文件完整内容，理解现有逻辑后再修改

## 四、代码质量自检（强制）

生成每段代码后必须逐项自检：

- [ ] **架构合规**：各层职责正确，无越层调用，依赖方向正确
- [ ] **数据安全**：参数化查询无SQL拼接，错误信息不暴露内部细节，输入有校验，无注入风险
- [ ] **数据完整性**：多表操作用事务，删主表清关联，唯一字段有约束
- [ ] **错误处理**：400/404/500 分类正确，无忽略的错误，分层传递
- [ ] **边界条件**：空值/nil/空数组处理，超长字符串和超大分页有限制
- [ ] **代码规范**：导出符号有文档注释，统一响应格式，命名遵循驼峰
- [ ] **性能**：无 N+1 查询，大数据量有分页且限制最大 PageSize
- [ ] **日志**：关键操作和错误记录了日志，日志无敏感字段
- [ ] **常见盲区**：无硬编码，Create 时设了必要默认值

## 五、索引自维护（强制）

AI 必须自主维护 `.ai/index.md`。以下情况必须在同一次回复中直接更新（禁止只说"记得更新"）：

| 触发场景                | 更新章节              |
| ----------------------- | --------------------- |
| 新增/删除文件目录       | 目录结构总览          |
| 新增/修改/删除 API 路由 | API 路由表            |
| 新增/修改数据模型       | 数据模型关系 + 表清单 |
| 新增/修改模块调用关系   | 模块调用关系          |

索引必须包含：目录结构总览、API 路由表、数据模型关系（含表清单）、模块调用关系。每次更新同步更新顶部时间戳。变更历史由 git log 追踪，索引中不记录。

## 六、编码规范（强制 · 可追加）

> ⚠️ **项目现有编码风格的一致性优先于本章规范。** 冲突时以项目现有风格为准。

### 通用原则

- 所有代码必须使用项目指定的格式化工具格式化
- 命名遵循项目现有风格（驼峰/蛇形），保持一致性
- 导出符号必须有文档注释
- 错误必须处理，禁止忽略
- 尽早返回，减少嵌套深度
- 禁止魔法数字，常量应集中管理
- 注释掉的代码提交前必须删除

### 语言特定规范

语言特定的详细编码规范（格式化、命名、控制结构、函数参数等）见 `rules/languages/` 目录下对应文件，AI 应根据项目语言按需加载：

- Go 语言：[code-style.md](../languages/go/code-style.md)
