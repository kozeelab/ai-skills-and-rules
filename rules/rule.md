# AI 协作开发规范

> **AI 必须严格遵守本文件中所有标注「强制」的规则，不得以任何理由跳过。**

---

## 一、强制执行协议

> ⚠️ **任何涉及代码修改的任务，必须先完成本协议，再动手写代码。没有"太简单不需要走流程"的例外。**

### 流程总览

```
首次加载 → Phase 0: 项目扫描 → 补全本文件中所有「待扫描」信息
                                    ↓
收到任务 → Phase 1: 启动检查 → Phase 2: 执行 → Phase 3: 收尾检查
              ↓ 必须输出           ↓ 写代码        ↓ 必须输出
           「📋 任务启动清单」                    「✅ 任务完成清单」
```

### Phase 0：项目扫描与信息补全

当「二、项目基本信息」或「三、架构约束」中包含 `[待扫描]` 标记时，**必须立即中断一切其他任务**，优先扫描并回填信息。

**扫描步骤**（步骤 1~4 可并行执行）：
1. 读取 `go.mod` 等依赖文件 → 识别语言版本、框架、ORM
2. `list_dir` 递归列出 `cmd/`、`internal/`、`api/` 等目录 → 识别分层架构
3. `grep_search` 搜索 `router` 等关键词 → 确认路由结构
4. 搜索 `.yaml`/`.toml` 等配置文件 → 识别配置管理和日志库
5. 抽样读取 2~3 个 Controller/Service/Model 文件 → 确认编码风格
6. 生成/更新 `.ai/index.md` 索引

**扫描完成后必须**：回填本文件所有 `[待扫描]` → 生成索引 → 用 `grep_search` 验证无残留 → 输出扫描报告。

```

━━━━━━━━━━━━━━
```

## 三、架构约束（强制）

### 3.1 分层结构

| 层                   | 文件位置                         | 职责                             | 禁止事项               |
| -------------------- | -------------------------------- | -------------------------------- | ---------------------- |
| Router               | [待扫描：如 internal/router/]    | 路由注册、中间件挂载             | 不写业务逻辑           |
| Controller / Handler | [待扫描：如 api/ 或 controller/] | 参数绑定、调用 Service、响应封装 | 不直接操作数据库       |
| Service              | [待扫描：如 internal/service/]   | 业务逻辑、数据转换               | 不直接操作 HTTP 上下文 |
| Repository / Model   | [待扫描：如 internal/model/]     | 纯数据库操作（CRUD）             | 不写业务逻辑           |

**依赖方向**：Router → Controller → Service → Repository，**严禁逆向依赖**。

### 3.2 强制规范

- 统一响应格式：所有 API 必须通过统一的响应工具函数返回，**禁止**直接拼装响应体
- 所有导出的结构体/DTO 字段必须有注释
- 新增 API 必须有完整的 Swagger / OpenAPI 文档注释
- 错误处理必须分层传递，禁止吞掉错误

## 四、上下文管理（强制）

> AI 在动手写代码之前，必须先建立充分的上下文。

### 4.1 索引优先原则

任何涉及"定位文件"的操作，**必须优先读取 `.ai/index.md`**，禁止盲目搜索。适用场景：修改已有功能、多模块改动、Bug 追踪、新增 API 确认路由冲突、新增表确认关联关系。

### 4.2 场景 → 主动读取参考文件

| 场景                  | 必须主动读取的文件                                        |
| --------------------- | --------------------------------------------------------- |
| **新增 API**          | 路由注册文件、同类 Handler/Service/Model、DTO/Entity 定义 |
| **修改现有接口**      | 目标 Handler、对应 Service/Model、相关 DTO                |
| **新增数据库表/字段** | Entity/Model 定义、DTO 文件、数据库初始化/迁移文件        |
| **Bug 修复**          | 出问题的文件、调用方和被调用方                            |
| **重构**              | 所有受影响的文件、调用方文件                              |

### 4.3 行为约束

- **禁止猜测**：缺少关键参考文件时，禁止凭空猜测字段名/类型/结构体定义/函数存在性/表结构
- **禁止过度读取**：不要一次性读取超过 8 个文件，分阶段按需读取
- **必须参考范本**：生成新代码时，必须找到同类实体的现有实现作为范本，禁止从零凭空生成
- **先读后改**：修改已有功能时，必须先读取目标文件完整内容，理解现有逻辑后再修改

## 五、代码质量自检（强制）

生成每段代码后必须逐项自检：

- [ ] **架构合规**：各层职责正确，无越层调用，依赖方向正确
- [ ] **数据安全**：参数化查询无SQL拼接，错误信息不暴露内部细节，输入有校验，无注入风险
- [ ] **数据完整性**：多表操作用事务，删主表清关联，唯一字段有约束
- [ ] **错误处理**：400/404/500 分类正确，无忽略的错误，分层传递
- [ ] **边界条件**：空值/nil/空数组处理，超长字符串和超大分页有限制
- [ ] **代码规范**：导出符号有文档注释，统一响应格式，命名遵循驼峰
- [ ] **性能**：无 N+1 查询，大数据量有分页且限制最大 PageSize
- [ ] **日志**：关键操作和错误记录了日志，日志无敏感字段
- [ ] **常见盲区**：无硬编码，Create 时设了必要默认值

## 六、索引自维护（强制）

AI 必须自主维护 `.ai/index.md`。以下情况必须在同一次回复中直接更新（禁止只说"记得更新"）：

| 触发场景                | 更新章节              |
| ----------------------- | --------------------- |
| Phase 0 首次扫描        | 全部章节              |
| 新增/删除文件目录       | 目录结构总览          |
| 新增/修改/删除 API 路由 | API 路由表            |
| 新增/修改数据模型       | 数据模型关系 + 表清单 |
| 新增/修改模块调用关系   | 模块调用关系          |
| 任何代码变更            | 变更日志（追加记录）  |

索引必须包含：目录结构总览、API 路由表、数据模型关系（含表清单）、模块调用关系、变更日志。每次更新同步更新顶部时间戳。

## 七、编码规范（强制 · 可追加）

> ⚠️ **项目现有编码风格的一致性优先于本章规范。** 冲突时以项目现有风格为准, 并尽量遵守本章编码规范。

### 7.1 格式化与代码限制（必须）

- 所有代码必须使用 `gofmt` + `goimports` 格式化
- 建议一行不超过 120 列（函数签名、struct tag、长字符串等例外）
- 文件 ≤ 800 行，函数 ≤ 80 行；单测文件 ≤ 1600 行，单测函数 ≤ 160 行
- Import 分组：标准库 → 内部包 → 第三方包，按字母排序，禁止相对路径

### 7.2 命名规范（必须）

- **文件名**：小写 + 下划线（如 `agent_service.go`）
- **包名**：小写单词，与目录一致，禁止 `util`/`common` 等无意义包名
- **结构体**：驼峰，名词短语，禁止 `Data`/`Info` 等宽泛名称，多行声明和初始化
- **变量/常量**：驼峰式，私有小写开头；特有名词如 `HTTPClient`、`userID`、`APIClient`
- **常量**：枚举类型先创建类型；驼峰式，禁止全大写+下划线
- **函数**：驼峰式；接口单函数以 `er` 后缀
- **方法接收器**：类名首字母小写，禁止 `me`/`this`/`self`

### 7.3 错误处理（必须）

- error 必须是最后一个返回参数，必须处理不得忽略
- 采用独立错误流，禁止与其他变量组合判断
- 禁止 panic 做一般错误处理（init/全局变量初始化例外）
- recover 必须在 defer 中使用，禁止滥用
- 禁止使用 `fmt.Errorf()` 返回错误，必须使用 `errno` 包自定义错误
- 错误必须立即用 logger 打印

```go
// ✅ 正确
if err != nil {
    logger.Error(ctx, "获取服务失败", zap.String("id", id), zap.Error(err))
    return nil, errno.ErrMCPServerNotFound
}

// ❌ 错误
return nil, fmt.Errorf("get server failed: %v", err)
```

### 7.4 注释规范（必须）

- 每个导出的类型/函数/方法/常量/变量必须有文档注释
- 格式：`// 符号名 描述信息`
- 包注释格式：`// Package 包名 描述`（main 包除外）
- 结构体内导出成员如意义不明确需注释
- 注释掉的代码提交前必须删除（或附注释说明保留原因）

### 7.5 控制结构（必须）

- if：变量在左常量在右，bool 直接判断
- range：只需 key 时省略第二个值，只需 value 时 `_, v`
- switch：必须有 default
- 禁止 goto（业务代码）
- 尽早 return，嵌套深度不超过 4 层
- 禁止在循环中使用 defer
- defer 必须在错误检查之后

### 7.6 函数与参数（推荐）

- 参数数量不超过 4 个（含 ctx），过多封装为结构体
- 尽量值传递；map/slice/chan/interface 不传指针
- 变量声明遵循就近原则
- 禁止魔法数字，常量放 `internal/constant` 目录下

### 7.7 RESTful API 规范（必须）

- URL 使用名词复数，HTTP Method 区分语义（POST 创建、GET 查询、PUT 更新、DELETE 删除）
- 路径段小写 + 中划线，版本前缀 `/api/v1/`
- 所有路由集中在 `internal/router/router.go` 注册
- 按资源类型分组，鉴权通过中间件挂载

### 7.8 HTTP 状态码与统一响应（必须）

- 严格按语义返回：200/201/400/401/403/404/409/500
- **必须**使用 `util.Success/Created/BadRequest/NotFound/InternalError/HandleError` 等统一响应函数
- **严禁** `c.JSON()` 直接拼装响应体
- 业务错误通过 `errno` 包定义，`util.HandleError(c, err)` 自动映射状态码

### 7.9 请求参数绑定与校验（必须）

- JSON Body 用 `c.ShouldBindJSON`，Query 用 `c.ShouldBindQuery`，路径用 `c.Param`
- 禁止 `c.Bind()`；DTO 必须使用 `binding` tag 校验
- 绑定失败立即 `util.BadRequest(c, err.Error())` 并 return
- 路径参数获取后必须校验非空

### 7.10 分页查询（必须）

- 列表接口嵌入 `entity.PageQuery`，Query 传参 `page`/`page_size`
- 默认 page=1, page_size=20，最大 100
- 返回 `entity.PageResult`（total/page/page_size/items）
- 排序字段通过 `SafeOrderBy` 白名单校验

### 7.11 Handler 层代码结构（必须）

- 标准流程：① 参数绑定 → ② 调用 Service → ③ 返回响应
- Handler 禁止包含业务逻辑、禁止直接操作数据库
- 调用 Service 时传递 `c.Request.Context()`

```go
// ✅ 正确
func (a *AgentAPI) Create(c *gin.Context) {
    var req entity.CreateAgentReq
    if err := c.ShouldBindJSON(&req); err != nil {
        util.BadRequest(c, err.Error())
        return
    }
    agent, err := a.agentService.CreateAgent(c.Request.Context(), &req)
    if err != nil {
        util.HandleError(c, err)
        return
    }
    util.Created(c, agent)
}
```

### 7.12 DTO 设计规范（必须）

- Create DTO：值类型 + `binding:"required"`
- Update DTO：指针类型（区分不传和清空）
- 请求/响应 DTO 分离，统一放 `internal/entity/`
- 每个字段必须有 `json` tag 和注释（含义+枚举值+默认值+约束）

### 7.13 Swagger 文档注释（必须）

- 新增 API 必须包含完整注释：`@Summary` `@Description` `@Tags` `@Accept` `@Produce` `@Param` `@Success` `@Failure` `@Router`
- 接口变更时同步更新注释

### 7.14 中间件规范（必须）

- 全局中间件在 `router.go` 统一注册，鉴权通过路由分组挂载
- 中间件职责单一，上下文信息用 `c.Set()`/`c.Get()` 传递
- 错误必须 `c.Abort()` 终止请求链

### 7.15 安全规范（必须）

- 所有数据库操作使用 GORM 参数化查询，严禁 SQL 拼接
- 用户输入必须校验长度/格式/范围
- 敏感信息（密码/Token/Secret Key）禁止出现在日志和错误响应中
- 密钥禁止硬编码，通过配置文件或环境变量管理

### 7.16 单元测试（必须）

- 文件名 `xxx_test.go`，函数名 `Test` 开头
- 每个重要可导出函数都要有测试用例
- Go 1.11+ 必须使用 go modules

### 7.17 推荐工具（可选）

- `gofmt` / `goimports` / `go vet` / `golint`
